Overview

This flow handles Microsoft 365 (GUID) group requests only. SharePoint site-group changes are applied directly by the web part (not via Graph).

SharePoint-direct changes still land in this list as Status = Completed audit rows. The flow must skip any non-Pending row (see the guard in §1) or it re-processes them, the Graph call fails on the SharePoint group's integer GroupId, and it overwrites Status to Failed — the user sees "Completed, then Failed."

1. Trigger & identity

  • Trigger: When an item is created on the Group Membership Requests list (the request/audit list).
  • Concurrency: Degree of Parallelism = 1 (serialize; avoids interleaving on the same group).
  • Runs as: a service account that is Owner of every managed O365 group, or an app registration with application permission GroupMember.ReadWrite.All. (The web part itself holds only read scopes — this identity is the elevation.)

FIRST ACTION (required) — guard on Status. If Status is not Pending, terminate immediately (do nothing / Cancelled). Only Status = Pending rows are real O365 work. Without this guard the flow re-processes the SharePoint-direct Completed audit rows and fails them. (Optionally also skip when GroupId is not a GUID, as belt-and-suspenders.)

Owner actions need a broader scope. Add Owner / Remove Owner (§5) act on the group's owners collection, which requires Group.ReadWrite.All (delegated or application) — GroupMember.ReadWrite.All covers members only. Add and admin-consent Group.ReadWrite.All on the flow identity before enabling owner requests, or those rows fail with an authorization error.

Never authorize on client-writable fields (RequestedBy/RequestedById, OfficeGroupRecordId, or GroupId as a trust anchor). Only Author/AuthorId is server-stamped and trustworthy.

3. Request item — fields the flow WRITES back

FieldValue

ResultMessageOutcome or error text.
AuthorizationCheckedYes/No.
AuthorizationResultAuthorization outcome text.

Invariant: no path may leave Status = Pending. Every branch (authorized, not-authorized, Graph error, exception) must end by writing a terminal Status.

4. Authorization re-check multi-value

The Group Management Authorized Admins list is one row per admin, and OfficeGroupRecord is a multi-value lookup into Managed Groups (an admin's whole group set is on one row). Authorize like this:

4a — resolve the requester's SharePoint user id

Read the created item's numeric AuthorId:

GET {listsSite}/_api/web/lists/getbytitle('Group Membership Requests')/items({ID})?$select=AuthorId

4b — read the requester's authorized groups (expanding the multi-value lookup)

GET {listsSite}/_api/web/lists/getbytitle('Group Management Authorized Admins')/items
    ?$select=Id,OfficeGroupRecord/GroupId
    &$expand=OfficeGroupRecord
    &$filter=UserId eq {AuthorId}
Header:  Accept: application/json;odata=nometadata

Filter by UserId (the single-value User Person column — filterable, indexed), not by the group. SharePoint can't reliably $filter a multi-value lookup's sub-field, so you fetch the requester's row(s) and test group membership inside the flow.

4c — test membership

Build the set of all OfficeGroupRecord[].GroupId values across the returned row(s). The request is authorized iff the request item's GroupId is in that set (case-insensitive GUID compare). Under odata=nometadata, OfficeGroupRecord is a plain array per row; under odata=verbose it is { "results": [ … ] }.

  • Authorized → proceed to §5; set AuthorizationChecked = Yes, AuthorizationResult = "Authorized".
  • Not authorized (group not in the set, or no rows) → fail closed: AuthorizationChecked = Yes, AuthorizationResult = "Requester not authorized for this group", Status = Failed, clear ResultMessage. Make no Graph call.

5. Perform the change (only if authorized)

Using GroupId (target group) and MemberId (the person's object id):

Add Member:
POST https://graph.microsoft.com/v1.0/groups/{GroupId}/members/$ref
Body: { "@odata.id": "https://graph.microsoft.com/v1.0/directoryObjects/{MemberId}" }
  → treat HTTP 400 "already exists" as success (idempotent)

Remove Member:
DELETE https://graph.microsoft.com/v1.0/groups/{GroupId}/members/{MemberId}/$ref
  → treat HTTP 404 "not found" as success (already not a member)

Add Owner:   (needs Group.ReadWrite.All)
POST https://graph.microsoft.com/v1.0/groups/{GroupId}/owners/$ref
Body: { "@odata.id": "https://graph.microsoft.com/v1.0/directoryObjects/{MemberId}" }
  → treat HTTP 400 "already exists" as success
  (an owner should also be a member — optionally also POST to /members/$ref)

Remove Owner:   (needs Group.ReadWrite.All)
  GUARD FIRST: GET .../groups/{GroupId}/owners?$count=true   (header ConsistencyLevel: eventual)
    if only ONE owner → Status = Failed, ResultMessage = "Cannot remove the last owner", no change
DELETE https://graph.microsoft.com/v1.0/groups/{GroupId}/owners/{MemberId}/$ref
  → treat HTTP 404 "not found" as success (already not an owner)

Never leave a group ownerless. Run the last-owner count before the Remove Owner DELETE, and refuse when it would drop the final owner.

On any other Graph error → Status = Failed, ResultMessage = the Graph error. On success → Status = Completed, ResultMessage = e.g. "Added" / "Removed".

6. Invariants (contract the web part relies on)

  1. Authorize on AuthorId only (server-stamped Created By); ignore all client-writable fields for security.
  2. Fail closed — any error, missing row, or ambiguity → Status = Failed, no Graph call.
  3. Exact Status casing — Completed / Failed (never Complete, Success, etc.).
  4. No row left Pending — every branch writes a terminal status. (A scheduled "sweep" flow that fails stale Pending rows older than N minutes is recommended as a backstop.)
  5. Idempotent — tolerate already-member / not-member (see §5).
  6. Only the flow identity should be able to edit Status / Authorization* (list permissions).

7. Flow work outstanding — owner add/remove web part v1.10.0

Owner management (needs a new Graph permission). The web part files Add Owner and Remove Owner requests (O365 groups only; shipped in the web part at v1.10.0, flow side still to do). To make them work the flow must:

  • Handle the two new Action values (§2) with the group owners Graph calls in §5.
  • Run the last-owner guard before every Remove Owner (§5) — never leave a group ownerless.
  • Use Group.ReadWrite.All for owner calls — GroupMember.ReadWrite.All covers members only (§1). Add and admin-consent it before enabling owner requests.
  • Member add/remove branches are unchanged; owner requests reach the flow through the same trigger, carry the same fields (§2), and write back the same way (§3) — only the perform-change branch and permission differ.

Verified against the web part. The web part writes the Action value verbatim, so the flow's switch matches on the exact strings Add Owner / Remove Owner. Owner and member requests are the identical payload — only the Action string differs — so there is no new field for the flow to read.

Still required from v1.8.0 — the multi-value authorization re-check (§4). The list's multi-value OfficeGroupRecord column and the flow's §4 check must change together. The web part reads both list layouts, so the list can migrate gradually.

No flow change for v1.11.0. v1.11.0 is the web part's self-diagnosing list-health check — it validates the SharePoint list schema and shows setup guidance to page editors. It reads list metadata only and needs nothing from the flow.

8. Nightly Group Site Permissions sync optional, separate flow

A scheduled flow can keep the Group Site Permissions list current instead of hand-maintaining it, by discovering which sites each managed group has permissions on. This is a separate flow from the request flow above.

  • Identity & scope. A dedicated app registration with Sites.Selected, using an app-only bearer token (gov endpoints login.microsoftonline.us / *.sharepoint.us) passed as an explicit Authorization header on raw SharePoint REST calls. Scan a curated "Managed Sites" list, never the whole tenant.
  • Sync-only columns (the web part ignores them): Source (Choice Auto|Manual), SyncKey, LastSyncedRunId, SyncStatus, StaleRuns, LastSyncedUtc.
  • Per site (root web): read roleAssignments ($expand Member, RoleDefinitionBindings); match group principals to Managed Groups GroupIds by claim prefix (c:0t.c|tenant| = security group, c:0o.c|federateddirectoryclaimprovider| = M365, strip trailing _o); for each PrincipalType=8 SharePoint site group, also read sitegroups(<id>)/users (most grants are nested); Permission = highest role, dropping "Limited Access"; diff vs existing Auto rows and upsert via $batch.
  • Reconcile (soft delete). Unstamped Auto rows → SyncStatus=Stale, StaleRuns++; hard-delete only at StaleRuns >= 3. Abort if any site GET failed or the stale set exceeds ~50%. Every query/write/delete is filtered Source eq 'Auto', so hand-entered Manual rows are never touched.
  • Known limits: site root web only; Site Collection Admins do not appear in role assignments; one level of group nesting.

See Setup-Guide.html → "Nightly sync" for the full walkthrough.
